OpenSpec · align-pi-update-with-resolved-pi

Update the pi wired to the dashboard — by delegating to its own pi update

Stop re-implementing the update. Resolve pi once, run that pi's built-in updater, surface that pi's version.

The key realisation

pi already owns install-location logic. Its CLI exports detectInstallMethod(), getSelfUpdateCommand(), and getSelfUpdateUnavailableInstruction(), pins the exact checked version (getSelfUpdatePlan), handles the @mariozechner→@earendil-works scope migration, and already refuses unsupported installs: "pi cannot self-update this installation."

So the dashboard should not classify install trees or shell out to npm install -g itself. It should invoke the resolved pi's own pi update and relay the result.

The data flow

┌──────────────────────────────────────────────────────────────┐ │ RESOLVE ONCE — the existing spawn source of truth │ │ resolvePi() → ToolRegistry.resolveExecutor("pi").argv │ │ realpath → pkgRoot → read package.json → version, path │ └───────────────────────────┬────────────────────────────────────┘ │ the SAME argv used to spawn sessions ┌───────────────┼────────────────────┐ ▼ ▼ ▼ ── STATS / HEALTH ── ── UPDATE ── ── CHECK ── wiredPi:{version, spawn: ask pi for latest path, method, <argv> update via its own update updatable} --self (or --all check (pinned ver, shown in Settings to fix extensions) scope-migration aware) │ ▼ pi runs detectInstallMethod(): npm|pnpm|bun|managed → builds correct self-update cmd, runs it unsupported (source/electron/brew) → prints the manual instruction + exits non-zero │ ┌───────────────┴────────────────┐ ▼ ▼ exit 0: stream output, non-zero: capture stderr instruction, then POST /api/restart show it on the row, button stays disabled (+ reload sessions) (no false success)

What the dashboard does (thin wrapper)

1
Resolve once. resolveWiredPi() returns { argv, pkgRoot, version, path } from ToolRegistry.resolveExecutor("pi") — identical to the spawn path. One source of truth for spawn + stats + update.
2
Stats. /api/health (and the Settings → Pi Ecosystem row) report wiredPi: { version, path } read from pkgRoot/package.json — never a separate npm list enumeration.
3
Update = delegate. Replace defaultRunNpmUpdate's npm install for the pi row with spawn(argv[0], [...argv.slice(1), "update", "--self"]) (or "--all"). Stream stdout/stderr to the existing progress events. pi decides method + does it.
4
Honest refusal for free. When pi exits non-zero with getSelfUpdateUnavailableInstruction() text (source checkout, electron bundle, brew, …), the dashboard shows that exact instruction. No dashboard-side install-kind table to maintain.
5
Restart after success. On exit 0, POST /api/restart + reload sessions so the new pi is actually picked up (the running server resolved the old binary).

What stays npm-based: the dashboard package itself

@blackbelt-technology/pi-agent-dashboard is not pi — there is no pi update for it. It keeps an npm/refuse path, but driven by the already-existing detectInstallLayout() + suggestedReinstallCommand() in recovery-server.ts (electron → reinstall app · monorepo → npm install from repo root · npm-global → npm install -g). Same "update the real tree, else instruct" contract, smaller surface.

Why this is the right shape

  • Can't target the wrong tree — you invoke the resolved pi; it updates itself.
  • Zero duplicated logic — install-method detection, version pinning, scope migration, Windows quarantine, and refusal all live in pi and are reused, not re-implemented.
  • Fixes extension skew too — --all updates pi + extensions in one consistent step (the exact failure that broke session spawning).
  • Stats and updates agree — both read/act on the one resolved path.

Open decision

--self (pi only) vs --all (pi + extensions) as the dashboard's default button action. --all would have prevented this incident, but it also updates every extension on a single click — more surface, longer run. Likely: button = --self; a separate "Update pi & extensions" action = --all.
