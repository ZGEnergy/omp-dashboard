## Why

The dashboard `packages/flows-plugin/` was built for an old pi-flows model (flow-architect subagent + `conditional`/`agent-loop-decision` step types). Seven recent pi-flows changes invalidated large parts of it: flow-architect was deleted (its `flow:architect-*` events are never emitted again), authoring moved to plain main-session tool calls (`flow_write` / `flow_agents`), new `code` and `code-decision` step types arrived, the failure model gained soft/hard outcomes, agents gained a validated outputs contract, and an edit-mode toggle is now driven by an inbound `flow:set-edit-mode` event the dashboard is expected to emit. The plugin currently renders dead architect UI and cannot render the new node types, outputs, decision branches, or authoring tool calls.

## What Changes

- **BREAKING (removal): delete all flow-architect surface.** Remove `architect-reducer.ts`, `FlowArchitect*.tsx`, `ArchitectInputPrompt`, the two architect slot claims, and the 14 `architect_*` entries from `FLOW_EVENT_MAP`. pi-flows no longer emits any `architect_*` / `flow:architect-*` event.
- **Two render paths, made explicit.** Authoring tool calls (`flow_write`, `flow_agents`) run in the main session and arrive as ordinary tool-call timeline entries → rendered via a **tool-renderer slot**. Running steps continue through `FLOW_EVENT_MAP` → flow-reducer → flow card grid.
- **Event map + reducer consume `nodeKind`.** Per the pi-flows `surface-node-kind` change, every node emits a first-class `nodeKind` (`agent` | `agent-decision` | `code` | `code-decision` | `fork` | `flow-ref`) on its lifecycle events, forwarded through the FlowManager seam and carried inside the event `data`. `FLOW_EVENT_MAP` preserves it and the reducer reads it (decided once at `started`), plus the soft/hard failure outcome and per-step typed `outputs`.
- **New running step cards.** A `code` card and a `code-decision` card (chosen branch + `↻ n/max` loop pill), an `outputs` KV section on both agent and code/code-decision cards, and distinct soft (amber, "routed → onError") vs hard (red, "halted flow") failure states.
- **Minimal live FlowGraph, extended.** Keep the existing React/dagre `FlowGraph` minimal (node names + running-step highlight; details live in the cards), adding `code`/`code-decision` node shapes and backward/loop edges (`max_iterations`). Remove the dead `conditional`/`agent-loop-decision`/`loopTarget` mappings.
- **Authoring tool-renderers.** Tool-renderer claims for `flow_write` and `flow_agents` that render from the real tool contract (`{written,name,namespace,command,path,diagnostics[]}` / catalog array). The `flow_write` success card renders a **static Mermaid snapshot** of the flow graph (via the `ui:markdown-content` primitive) and step/agent counts, both **parsed client-side from the tool args** (the submitted YAML). The "view file/YAML" sub-row shows the tool args. Validation diagnostics drive the success/error states. A **New/Edit launcher dialog** mirrors the OpenSpec buttons: it builds `/skill:edit-flow [name]` and fires the shared `onSendPrompt` prop (never authors directly).
- **Edit-mode settings.** A **global** dashboard plugin-config default (automation-plugin pattern: `usePluginConfig` + `plugin_config_write` + `useSettingsDraftSource`). When a session's flows plugin becomes available (via the existing `flowsAvailability` / `flow:rediscover` signal), the dashboard auto-reconciles that session by emitting `flow:set-edit-mode { enabled: <globalDefault> }`, which pi-flows persists to the project's local `.pi/settings.json`. An optional per-session override toggle stays on the flows subcard; the "New/Edit…" action is gated on edit-mode being on.
- **Subcard rework.** `SessionFlowActions` drops the removed `/flows:new` and `/flows:edit` command dependencies; offers Run / New-or-Edit (gated) / Delete / Abort.

## Capabilities

### New Capabilities
- `flow-authoring-renderers`: Tool-renderer slot claims for `flow_write` and `flow_agents`; rendering from the real tool result contract; client-side YAML parsing of tool args for the Mermaid snapshot + counts; diagnostics-driven success/error states; the args-backed "view file" sub-row; and the New/Edit launcher dialog that issues `/skill:edit-flow [name]` via `onSendPrompt`.
- `flows-edit-mode-settings`: The global plugin-config edit-mode default, the auto-reconcile-to-session emission of `flow:set-edit-mode` on flows-plugin availability, the optional per-session override toggle, and edit-mode gating of the New/Edit action.

### Modified Capabilities
- `flow-event-bridge`: `FLOW_EVENT_MAP` preserves the `nodeKind` discriminator (all 6 node types) and the soft/hard failure outcome on lifecycle events; the 14 `architect_*` mappings are REMOVED.
- `flow-card-status`: New `code` and `code-decision` step cards, the `outputs` KV section on agent + code/code-decision cards, soft-vs-hard failure visuals, the code-decision chosen-branch + loop-pill rendering, and the minimal FlowGraph node shapes/backward edges for the new node set (removing `conditional`/`agent-loop-decision`/`loopTarget`).
- `flow-architect-view`: REMOVED — the architect content-view capability is eliminated (no producer events remain).
- `flow-architect-popout`: REMOVED — the architect popout route/claim is eliminated.
- `flows-plugin`: FLOWS subcard availability (`shouldRenderFlowsSubcard` / `computeAvailability`) gated on pi-flows extension presence in the cwd (a `flows`/`flows:*` command in `commandsList`) OR a flow event — NOT `flowsList` length; the dead `flows:new` signal removed. Shows the subcard for active-but-empty flows cwds (the edit-mode/author-first-flow case).

## Impact

- **Code (dashboard `packages/flows-plugin/`):** delete `architect-reducer.ts`, `FlowArchitect.tsx`, `FlowArchitectPopoutPage.tsx`, `FlowArchitectPopoutClaim.tsx`, `ArchitectInputPrompt`, `architect-reducer.test.ts`, `ArchitectInputPrompt.test.tsx`; rework `flow-reducer.ts`, `FlowAgentCard.tsx`, `FlowAgentDetail.tsx`, `FlowGraph.tsx`, `SessionFlowActions.tsx`, `src/client/index.tsx` (claims), `package.json` (manifest claims: add `tool-renderer` + `settings-section`, drop architect claims); add authoring tool-renderer components, an edit-mode settings component, and a New/Edit launcher dialog.
- **Code (dashboard `packages/extension/`):** `flow-event-wiring.ts` — `FLOW_EVENT_MAP` preserves `nodeKind`/outcome passthrough, drops architect entries; outbound `flow:set-edit-mode` emission path.
- **Cross-repo (resolved by `surface-node-kind`):** the pi-flows `surface-node-kind` change finalizes the `nodeKind` contract (every executor emits it, forwarded through FlowManager, carried in `data`) and keeps the chosen `branch` in completion `typedOutputs`. This dashboard change is reducer-side only and lands independently; it reads `nodeKind` when present and falls back to an agent card for runs persisted before `surface-node-kind`.
- **Dependencies:** reuses existing dashboard runtime APIs — `tool-renderer` slot, `ui:markdown-content` (Mermaid fences), `usePluginConfig`/`plugin_config_write`/`useSettingsDraftSource`. No new external dependency.
- **Backward compatibility:** breaking for any consumer of architect events/claims (all dead upstream). Old persisted flow runs predate the new model; replay of architect runs is dropped intentionally.
