## Why

The first-launch chat-display preset modal (`FirstLaunchDisplayModal`, shipped by archived `configurable-chat-display`) becomes **permanently stuck on mobile**: the "How much should the chat view show?" question appears and neither **Skip** nor **Continue** dismisses it. Diagnosed against current code (`packages/client/src/components/FirstLaunchDisplayModal.tsx`, `packages/client/src/App.tsx`, `packages/server/src/browser-gateway.ts`, `packages/server/src/routes/preferences-display-routes.ts`).

The modal renders while `displayPrefsLoaded === true && displayPrefs === undefined` (App.tsx:1890). Its `onClose` is a **deliberate no-op** (App.tsx:1893) — closing is 100% dependent on `displayPrefs` flipping to a defined value. After mount, the client learns that value from exactly two sources: the one-shot mount `GET /api/preferences/display`, and live `display_prefs_updated` WS broadcasts. `seed()` in the modal PATCHes the preset, then calls the no-op `onClose`, and **discards the PATCH 200 body `{ displayPrefs: merged }`**. So a tap closes the modal only if the fire-and-forget WS broadcast lands back in this exact socket.

**RC2 is the primary, sufficient fix; RC1 and RC3 are resilience/hardening, not co-equal root causes.** A doubt-review corrected the earlier framing: RC1 cannot rescue the stuck modal in the case that matters most on mobile (a PATCH that never completes seeds nothing, so there is no pref to snapshot on reconnect), and even when the PATCH succeeds RC1 only helps *after* a reconnect round-trip — never instantly. Only RC2 (close from local state, no network dependency) dismisses the modal the moment the user taps.

Three contributing causes:

- **RC1 — no connect-time snapshot** (reconnect-resilience + cross-tab, not the stuck-modal fix)**.** On every (re)connect, `wss.on("connection")` re-pushes `sessions_snapshot`, `pinned_dirs_updated`, `favorite_models_updated`, `workspaces_updated`, `openspec_update`, `terminal_added` — but **never `display_prefs_updated`** (browser-gateway.ts:340-355). Display-prefs is the only preference with no connect snapshot, so a missed broadcast is unrecoverable without a full page reload. `broadcast()`'s `fanout()` silently `continue`s past any socket whose `readyState !== OPEN` and never replays (browser-gateway.ts:296-301).
- **RC2 — no optimistic local close (PRIMARY).** `onClose` is a no-op and `seed()` throws away the PATCH response body; the modal waits on a broadcast it may never receive instead of closing from the value it already has in hand. (The swallowing `catch { /* broadcast will reconcile */ }` at `FirstLaunchDisplayModal.tsx:43` is itself false when the socket is not OPEN — the broadcast is *not* reconciled — and is the smell that seeded the bug.)
- **RC3 — failed GET conflated with "no prefs."** The mount fetch does `if (!r.ok) return;` then `setDisplayPrefsLoaded(true)` in `finally` (App.tsx:709-716). A 403/flap leaves `displayPrefs === undefined` + `loaded === true` → the modal opens **spuriously**, not just on a genuine first launch.

Mobile is exactly where the WS socket is not reliably `OPEN` at the instant of the PATCH: the mount-time WS handshake races a fast tap, and mobile browsers suspend/close WebSockets on tab-backgrounding or screen-lock, reconnecting later (backoff climbs to 30s, `useWebSocket.ts`) with no replay. The result: the PATCH persists server-side, but the client never learns, `displayPrefs` stays `undefined` forever, and the no-op `onClose` provides no fallback.

**Testable repro (acceptance oracle):** with the modal stuck, a full page reload dismisses it — because the PATCH already persisted; only the live notification was lost.

## What Changes

- **MODIFY** `packages/client/src/components/FirstLaunchDisplayModal.tsx` — `seed()` SHALL **unconditionally** apply the chosen prefs locally so the modal closes immediately, independent of any WS broadcast **and independent of whether the PATCH succeeds or fails**. The applied value SHALL be `DISPLAY_PRESETS[key]`; the PATCH 200 response body `{ displayPrefs: merged }`, when readable, MAY *refine* it. `onClose` MUST fire on every path (success, non-2xx, thrown fetch) — matching the current code's unconditional close — so a failed PATCH still dismisses the modal locally rather than stranding it open. Remove the misleading `catch` comment. (RC2)
- **MODIFY** `packages/client/src/App.tsx` — the `FirstLaunchDisplayModal` `onClose` SHALL actually close the modal (seed `displayPrefs` locally) rather than being a no-op. For RC3, introduce a distinct `displayPrefsSeedless` flag set true ONLY when `r.ok && body.displayPrefs === undefined` (needed because `setDisplayPrefsLoaded(true)` lives in the fetch's `finally` and runs even after `if (!r.ok) return`, so `loaded && undefined` is true on ANY failed GET). **The modal render gate SHALL be `displayPrefsSeedless && displayPrefs === undefined`** — NOT the seedless flag alone. Keeping the `&& displayPrefs === undefined` term is load-bearing (cross-model doubt-review): it is what lets `onClose`→`setDisplayPrefs(prefs)`, a cross-tab `display_prefs_updated` broadcast, AND the RC1 connect snapshot each close the modal by defining `displayPrefs`; a seedless-only gate would leave the modal open after all three. (RC2, RC3)
- **MODIFY** `packages/server/src/browser-gateway.ts` — the `wss.on("connection")` handshake SHALL send a `display_prefs_updated` snapshot on connect, guarded like the sibling snapshots (`typeof preferencesStore.getDisplayPrefs === "function"` to match the adjacent `getFavoriteModels`/`getWorkspaces` stub-safety guards) AND only when `getDisplayPrefs()` returns defined, so a client that missed a live broadcast recovers the pref on reconnect without a page reload. (RC1 — reconnect-resilience, not the stuck-modal fix)
- **DOCUMENTATION** — delegated per Rule 6 (caveman style): update the `chat-display-preferences` behavior notes in `docs/architecture.md` (connect-time snapshot parity + optimistic-close contract); per-file rows unchanged (no new files).

## In scope: desktop render gap (folded in per maintainer decision)

The `FirstLaunchDisplayModal` currently renders at **exactly one** site — `App.tsx:1891`, inside the `if (isMobile) { return (…) }` branch (opens at `App.tsx:1866`). The desktop layout (`App.tsx:2006+`) has **no** modal render, so the first-launch preset picker is **mobile-only** and desktop users are never prompted — a pre-existing gap contradicting the shipped spec's unqualified "the client MUST display a one-shot modal". This is now **in scope** for this change:

- **MODIFY** `packages/client/src/App.tsx` — render the gated `FirstLaunchDisplayModal` in the desktop return as well as the mobile return. To stay DRY (avoid two divergent copies of the render + `onClose` wiring), extract a single `firstLaunchModal` element computed once from the shared gate (`displayPrefsSeedless && displayPrefs === undefined`) and render it in **both** the `if (isMobile)` return and the desktop return. `DialogPortal` already portals to `document.body`, so the mount location within each tree is immaterial — it only needs to be inside `apiProvider` so context is available.

## Impact

- Affected specs: `chat-display-preferences` (MODIFIED — first-launch dismissal made round-trip-independent; connect-time snapshot added; failed-GET no longer triggers the modal).
- Affected code: `packages/client` (modal + App wiring) and `packages/server` (one connect-handshake send). No new endpoint, no protocol version bump — `display_prefs_updated` and `PATCH /api/preferences/display` are unchanged; the connect snapshot reuses the existing message type.
- Behavior: **RC2 alone dismisses the stuck modal in every case** (WS OPEN or not, PATCH success or failure). RC1 does NOT fix the stuck modal — it adds reconnect-resilience/cross-tab parity so a seeded client that missed a live broadcast self-heals on reconnect. RC3 stops spurious opens on a failed GET. This change does all three: RC2 (required), RC1 (parity with sibling prefs), RC3 (hardening).
- No security/auth change: the connect snapshot rides the already-authenticated browser socket; no new write path.

## Discipline Skills

- `systematic-debugging` — the fix rests on a root-cause chain (missing connect snapshot + no-op onClose + failed-GET conflation) that must be verified by the reload-repro before and after, not patched by guess.
- `doubt-driven-review` — the connect-handshake change touches the WS protocol every browser depends on; confirm the added `display_prefs_updated` send does not regress the seedless-first-launch path (a genuine fresh install must still get `undefined` and see the modal exactly once).
