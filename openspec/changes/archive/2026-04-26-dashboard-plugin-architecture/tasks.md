- [x] 5.2 Confirm tree-shaking works for unused plugin React via test fixtures (one plugin with two unused exports — confirm they don't ship in the final bundle). **Done by `add-dashboard-shell-slots-runtime`** — the named-import emission shape (`import { Foo, Bar } from "plugin/client"`) was the explicit design choice precisely to enable Vite/Rollup tree-shaking of unused exports. - [x] 5.3 Prototype the `settings-section` slot end-to-end: a fake plugin contributes a React settings section, writes to its namespace, sees the broadcast, re-renders. Validates the persistence + reactive-broadcast loop before real plugins consume it. **Done by `add-dashboard-shell-slots-runtime`** — the demo plugin at `packages/demo-plugin/src/client.tsx` exports `DemoSettings` claiming `settings-section` (tab `"general"`). The full loop is wired:
  - **Slot consumer**: `SettingsSectionSlot` in `packages/dashboard-plugin-runtime/src/slot-consumers.tsx:173` filters claims by `tab` and renders contributions.
  - **Persistence**: `POST /api/config/plugins/:id` (`packages/server/src/routes/plugin-config-routes.ts`) validates against `configSchema` and merges into `~/.pi/dashboard/config.json#plugins.<id>`.
  - **Broadcast**: server emits `plugin_config_update { id, config }`; client subscribes in `plugin-context.tsx:235`.
  - **Reactive read**: `usePluginConfig<T>()` in `plugin-context.tsx:80` re-renders consumers within one frame.
  - **Tests**: `slot-consumers.test.tsx` covers tab filtering, default-to-general, no-claims-renders-nothing; `plugin-context.test.tsx` covers "re-renders on plugin_config_update" via `act()` and a fake WebSocket; `plugin-config-route.test.ts` covers the REST endpoint shape and status tracking.

Validates the seam before `extract-subagents-as-plugin` consumes it. **Done by `add-dashboard-shell-slots-runtime`** — the demo plugin at `packages/demo-plugin/src/client.tsx` exports `DemoToolRenderer` claiming `tool-renderer` for `toolName: "DashboardDemo"`. The slot consumer `ToolRendererSlot` (`slot-consumers.tsx:187`) uses `forToolName` to find a matching claim; if none, it falls through to a `FallbackComponent` (i.e. `GenericToolRenderer`). The synthetic tool name was renamed from `"Demo"` to `"DashboardDemo"` during runtime implementation (clearer disambiguation); semantically identical to the prototype task description.
