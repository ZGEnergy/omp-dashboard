## Why

The generated `openspec-*` skills invoke the CLI by **bare name** (`openspec
status …`). A pi session resolves bare names via `getShellEnv()`, which reads the
session process's live PATH. On a machine with no global `openspec` install (fresh
checkout on a branch — no worktree, no local dep on PATH), bare `openspec` does not
resolve, and the running agent silently degrades to hand-editing artifacts — a
lossy path the skill never authorizes.

The fix cannot live in the skills (regenerated by `npx @fission-ai/openspec init
--force`). It also cannot be a server-side check: the server bundles
`@fission-ai/openspec` as a dependency, so any server-side detection reports the CLI
"present" on the exact machine where the *session* cannot resolve it (verified:
`packages/server/package.json` pins it; `doctor-routes.ts:95 detectOpenSpecOnPath`
is PATH-only and reflects the human shell, not the session). The only durable fix is
to make bare `openspec` **resolve inside the session** — which only something
running in the session (the bridge extension) can guarantee.

## What Changes

- The bridge extension gains a pinned `@fission-ai/openspec` dependency and, at
  session start, ensures a dedicated **shim directory** containing an `openspec`
  executable that resolves to that pinned CLI.
- The bridge **prepends the shim directory to `process.env.PATH`** (idempotently),
  so `getShellEnv()` — which reads live `process.env` per command — hands every
  bash child (including the generated skills' bare `openspec`) a resolvable CLI.
- Result: bare `openspec` works in-session on any machine, offline, with a pinned
  version. No button gate, no server detection, no `npx` warm-up, no skill edits.
- **Single-source the openspec version** so the shim, the server poller, and the
  worktree skill-regen track ONE number (today the installed dep is 1.4.1 while
  skills read `generatedBy: 1.6.0` — real drift). Mechanism (revised after review
  — no root `overrides`, which would leak into the published root tarball and is
  redundant with npm's natural hoisting): (a) declare **compatible caret ranges**
  `^1.6.0` at the server dep AND the new extension dep — npm hoists them to ONE
  installed copy; (b) a **one-time force-regen** of the `openspec-*` skills at
  1.6.0, committed, so `generatedBy` matches the installed version; (c) harden
  worktreeInit to `npx --no-install openspec init --tools pi --force` (offline:
  errors instead of silently fetching if the local bin is missing — flags kept);
  (d) extend `verify-release-deps.mjs` to assert the server and extension ranges
  stay consistent, and wire it into `ci.yml` so drift is caught on `develop`, not
  only at release. The shim resolves *that one hoisted version*.

## Impact

- **Single source** (`1.6.0` via consistent `^1.6.0` ranges): server dep +
  extension dep both `^1.6.0` → npm hoists ONE installed copy; the extension dep
  exists because the shim `require.resolve`s it and must travel with the published
  extension into generic projects (the override would not, and would leak into the
  root tarball). `verify-release-deps.mjs` keeps the two ranges consistent + runs
  in CI. **Risk:** raising the installed version 1.4.1→1.6.0 moves the server
  poller too — verify `openspec status|list --json` AND `init --tools pi`
  flag-compat against 1.6.0, asserting the poller stays non-empty (it fails
  silent-empty on schema breaks) before this lands.
- **worktreeInit** (`.pi/settings.json`): keep `--tools pi --force`; add
  `--no-install` → `npx --no-install openspec init --tools pi --force` (offline
  hardening only; npx already resolves the local bin).
- **Extension** (`packages/extension/`): new bridge-init step that builds the shim dir + prepends it to
  `process.env.PATH` (the sole in-extension lever over the LLM Bash tool's child
  env — verified: `user_bash` fires only for interactive `!`, not the tool path).
  Cross-platform shim = an **extensionless** bash-resolvable script (Git Bash
  `bash.exe -c` ignores PATHEXT, so a `.cmd` would not resolve bare `openspec`),
  invoking the pinned bin via an **absolute** `node` path (`process.execPath`) so
  it survives a stripped system PATH.
- **No change** to: the generated skills, `SessionOpenSpecActions.tsx`, server
  detection, or the OpenSpec poll payload.
- **Scope of the fix**: addresses the *CLI-not-found* cause only. A corrupt shim
  or an unexpected install layout can still fail — in which case the bridge logs
  **and** emits a dashboard-visible `missingTool`-style signal (repo convention
  `register-bash-and-tool-install-help`), so the failure is seen, never silently
  hand-edited.
- **Install footprint**: adds `@fission-ai/openspec` (+ transitive `@inquirer/*`)
  to every extension install, including users who never touch OpenSpec — accepted
  (it is already a server dep) but noted.
- **Rollout**: the fix ships in the bridge; a machine benefits after its extension
  updates (inherent to any bridge-side fix).

## Discipline Skills

- `security-hardening`: the shim executes a bundled, lockfile-pinned binary (no
  `npx`-latest fetch, no postinstall RCE surface) — verify the shim cannot be
  hijacked (dir perms, no writable-by-others shim path) and PATH prepend order.
- `review-code`: bridge-init + cross-platform shim is non-trivial; review before commit.
