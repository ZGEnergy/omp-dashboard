## Why

The generated `openspec-*` skills invoke the CLI by **bare name** (`openspec
status â€¦`). A pi session resolves bare names via `getShellEnv()`, which reads the
session process's live PATH. On a machine with no global `openspec` install (fresh
checkout on a branch â€” no worktree, no local dep on PATH), bare `openspec` does not
resolve, and the running agent silently degrades to hand-editing artifacts â€” a
lossy path the skill never authorizes.

The fix cannot live in the skills (regenerated by `npx @fission-ai/openspec init
--force`). It also cannot be a server-side check: the server bundles
`@fission-ai/openspec` as a dependency, so any server-side detection reports the CLI
"present" on the exact machine where the *session* cannot resolve it (verified:
`packages/server/package.json` pins it; `doctor-routes.ts:95 detectOpenSpecOnPath`
is PATH-only and reflects the human shell, not the session). The only durable fix is
to make bare `openspec` **resolve inside the session** â€” which only something
running in the session (the bridge extension) can guarantee.

## What Changes

- The bridge extension gains a pinned `@fission-ai/openspec` dependency and, at
  session start, ensures a dedicated **shim directory** containing an `openspec`
  executable that resolves to that pinned CLI.
- The bridge **prepends the shim directory to `process.env.PATH`** (idempotently),
  so `getShellEnv()` â€” which reads live `process.env` per command â€” hands every
  bash child (including the generated skills' bare `openspec`) a resolvable CLI.
- Result: bare `openspec` works in-session on any machine, offline, with a pinned
  version. No button gate, no server detection, no `npx` warm-up, no skill edits.

## Impact

- **Extension** (`packages/extension/`): add `@fission-ai/openspec` as a dep pinned
  **exact `1.6.0`** â€” the version that generated the `openspec-*` skills
  (`generatedBy: "1.6.0"`), so the shim is flag-compatible with the exact commands
  those skills run. This intentionally diverges from the server's bundled 1.4.1
  (the poller runs its own copy; the two invocations are independent); new
  bridge-init step that builds the shim dir + prepends it to
  `process.env.PATH` (the sole in-extension lever over the LLM Bash tool's child
  env â€” verified: `user_bash` fires only for interactive `!`, not the tool path).
  Cross-platform shim = an **extensionless** bash-resolvable script (Git Bash
  `bash.exe -c` ignores PATHEXT, so a `.cmd` would not resolve bare `openspec`),
  invoking the pinned bin via an **absolute** `node` path (`process.execPath`) so
  it survives a stripped system PATH.
- **No change** to: the generated skills, `SessionOpenSpecActions.tsx`, server
  detection, or the OpenSpec poll payload.
- **Scope of the fix**: addresses the *CLI-not-found* cause only. A corrupt shim
  or an unexpected install layout can still fail â€” in which case the bridge logs
  **and** emits a dashboard-visible `missingTool`-style signal (repo convention
  `register-bash-and-tool-install-help`), so the failure is seen, never silently
  hand-edited.
- **Install footprint**: adds `@fission-ai/openspec` (+ transitive `@inquirer/*`)
  to every extension install, including users who never touch OpenSpec â€” accepted
  (it is already a server dep) but noted.
- **Rollout**: the fix ships in the bridge; a machine benefits after its extension
  updates (inherent to any bridge-side fix).
