# Design

## Context

- Generated `openspec-*` skills call bare `openspec` (regenerated by
  `openspec init --force` ŌåÆ not an editable surface).
- A pi bash command's env comes from `utils/shell.js getShellEnv()`, which reads
  **live `process.env`** each call and returns `{ ...process.env, PATH: <pi binDir
  prepended> }` (verified `shell.js:103-111`; it already prepends pi's own
  `getBinDir()` with a `hasBinDir` idempotency guard ŌĆö direct precedent).
- The Apply button fires the skill into a **live** session (no spawn-time PATH
  injection possible), and the failing machine has no global `openspec`.
- Server-side detection is the wrong layer: the server bundles
  `@fission-ai/openspec` (`packages/server/package.json`, locked 1.4.1), so a
  server probe says "present" precisely where the session cannot resolve it. The
  only component that knows the session's truth ŌĆö and can change it ŌĆö is the
  bridge, which runs inside the session.

## Decision: bridge prepends a pinned-CLI shim dir to `process.env.PATH`

```
 bridge init (session start)
   Ōöé  1. resolve pinned @fission-ai/openspec bin  (require.resolve, lockfile-pinned)
   Ōöé  2. ensure dedicated shim dir contains `openspec` ŌåÆ that bin  (idempotent)
   Ōöé  3. if shimDir Ōłē process.env.PATH ŌåÆ process.env.PATH = shimDir + delimiter + PATH
   Ō¢╝
 agent runs bash ŌåÆ getShellEnv() reads LIVE process.env ŌåÆ child sees shimDir
   Ō¢╝
 generated skill: `openspec status ŌĆ”`  ŌåÆ resolves the pinned CLI  Ō£ō  (no edit)
```

Why a **dedicated shim dir** rather than prepending the extension's
`node_modules/.bin`: `.bin` carries every transitive dep's executables; prepending
it would shadow unrelated binaries. A dir containing only an `openspec` shim scopes
the PATH change to exactly one name.

Why a **pinned dependency** rather than `npx -y @fission-ai/openspec`: the earlier
npx-warm-up design was rejected in review ŌĆö npx fetches *latest* (version skew),
runs postinstall scripts as the session user (RCE surface), needs network, and on
packaged Electron the stripped PATH can fail to resolve `npx` itself (exit 127). A
lockfile-pinned dep is offline, integrity-checked, and version-consistent with the
server's poller.

## Mechanism details (revised after review)

- **Bin resolution (F4)**: `@fission-ai/openspec` exports only `"."`, so
  `require.resolve("@fission-ai/openspec/bin/openspec.js")` throws
  `ERR_PACKAGE_PATH_NOT_EXPORTED`. Resolve the **package root** via
  `require.resolve("@fission-ai/openspec/package.json")`, then join
  `bin/openspec.js` from that dir. Caveat: the extension loads via jiti with a
  virtualModules alias map that does NOT cover `@fission-ai/openspec`, so
  resolution depends on a real adjacent `node_modules` entry ŌĆö which differs
  across published-npm / local-dev / global-pi installs (the
  `switch-extension-source` hazard). Fail soft: if resolution throws, skip the
  prepend and log ŌĆö never crash bridge init.
- **Absolute node, not bare `node` (F3)**: the resolved `bin/openspec.js` has a
  `#!/usr/bin/env node` shebang, so a shim that relies on `node` being on PATH
  fails under the same stripped-PATH condition we reject `npx` for. The shim
  therefore invokes `"<process.execPath>" "<bin.js>" "$@"` ŌĆö `process.execPath`
  is the absolute path to the node running the bridge, known at init. No
  `node`-on-PATH dependency.
- **Cross-platform shim is extensionless (F2)**: the bash tool runs through a
  shell ŌĆö on Windows that is Git Bash (`bash.exe -c`), which does NOT consult
  PATHEXT. A `.cmd` would never resolve bare `openspec`. So the shim is a single
  **extensionless** `#!/bin/sh` script (`exec "<execPath>" "<bin.js>" "$@"`),
  resolvable by the POSIX shell AND Git Bash. The repo's `.cmd` handling
  (`platform/exec.ts`) is irrelevant here ŌĆö it wraps Node `spawn()`, not a
  bash-child lookup.
- **Idempotency + canonicalization (F6)**: canonicalize the shim dir
  (`fs.realpathSync` ŌåÆ absolute, no `~`, no trailing slash) BEFORE the
  membership check; split `PATH` on `path.delimiter`, compare canonical forms,
  prepend only if absent. Survives `/reload` (guard like
  `dashboard-context-injector`'s `isActive`).
- **Atomic + refreshed write (F7, F8)**: write the shim to a temp file then
  `rename` (atomic across concurrent sessions). **Re-point on every init** (do
  not skip-if-exists) so an extension upgrade refreshes the `bin.js` target
  rather than leaving a stale/dangling shim.
- **Non-destructive, scoped (F10, F13)**: prepend only; the shim dir contains
  ONLY `openspec`, so it cannot shadow `node`/`git`/etc. that the bridge's own
  subprocesses rely on. The global `process.env.PATH` mutation does affect every
  later spawn in the pi process, but adds exactly one resolvable name. "Some
  `openspec` wins" is scoped to **global** PATH entries ŌĆö a project-local
  `node_modules/.bin/openspec` is not on the bash-child PATH regardless.
- **Version pin (F5, F14 ŌĆö resolved)**: pin **exact `1.6.0`**, the version that
  generated the `openspec-*` skills (`generatedBy: "1.6.0"`), because the shim's
  job is to run the exact commands those skills emit (`instructions apply
  --change --json`, `status --change --json`, `list --json`). This intentionally
  differs from the server's bundled 1.4.1: the poller runs the server's own copy
  and does not consume the session CLI's output, so there is no cross-process
  parity requirement ŌĆö flag-compat is with the SKILLS, not the server. (If
  worktreeInit's `npx ... init --force` later regenerates skills at a newer
  version, bumping this pin to match is the follow-up.)
- **Shim dir perms (security)**: stable per-user dir under `~/.pi/dashboard/`,
  created owner-only (`0700`) so a world-writable path can't hijack the shim; the
  shim TARGET is `require.resolve`d (not a PATH search), so the target is trusted.

## Goals / Non-Goals

- **Goal**: bare `openspec` resolves in-session on any machine, offline, pinned.
- **Non-Goal**: gating/altering the Apply button or any frontend (unchanged).
- **Non-Goal**: server-side detection or a warm-up endpoint (rejected in review).
- **Non-Goal**: fixing the upstream skill's bare-name call (out of our control).

## Risks

- **No in-extension fallback (F1, corrected)**: `getShellEnv()` reading live
  `process.env` is the ONLY lever over the LLM Bash tool's child env. The
  `user_bash` event fires only for the interactive `!`/`!!` TUI path (verified:
  emitted solely in `modes/interactive/interactive-mode.js`), NOT for the bash
  *tool* that runs skill commands ŌĆö so it is not a fallback. If a future pi
  version snapshots env at startup instead of reading it live, this approach
  breaks and the fix would require a pi-core change (exposing the bash-tool
  `spawnHook`/`operations` to extensions). Named honestly; there is no
  in-extension workaround.
- **Provisioning failure surfaces both ways (F9 ŌĆö resolved)**: on any failure
  (bin unresolvable, shim write error) the bridge **always logs** a diagnostic
  AND, on a hard failure, **emits a dashboard-visible `missingTool`-style
  payload** (repo convention `register-bash-and-tool-install-help`) so the user
  SEES that provisioning failed ŌĆö never a silent regression to hand-edit. Init
  still does not throw (fail-soft).
- **Rollout**: benefits a machine only after its bridge updates ŌĆö inherent to any
  bridge-side fix; documented, not mitigated.
- **Fix scope (F12)**: resolves the CLI-not-found cause only; a version-drift
  flag error or corrupt shim still errors normally (not hand-edited).
