Action selector — three interaction directions

All three are live: type in the search, expand groups, click an action — the payload form swaps to match. Same registry behind each (Core·2, Flows·4 with live cwd enum, Git·3, Slack disabled/not-configured). Pick the interaction model, then we polish that one.

Direction A

Inline accordion

Lives in the form. Groups collapse; search filters in place. Zero navigation.

Direction B

Command palette

A field opens a ⌘K-style overlay. Keyboard-first fuzzy search, collapses to a chip.

Direction C

Two-pane master/detail

Source rail on the left, that plugin's actions + payload on the right.

Read: A (accordion) is the safe default — lowest cost, best discoverability, already half-built. B (palette) wins hard once registrations pass ~20 and for keyboard-driven users, but hurts browse/discovery. C (two-pane) reads best at scale but is the worst on mobile and the priciest. A pragmatic path: ship A, and add B's search box as the same data source so it can graduate to a palette later without a rewrite. Try all three above before deciding.
