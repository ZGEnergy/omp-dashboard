Feeding dynamic inputs to a flow — via an automation

Mockup + annotated data flow. The automation carries one per-fire value (the file that arrived) into the flow's task using a generic ${{trigger.*}} variable channel. Any trigger fills the bag; any action's payload consumes it; the UI shows a palette of whatever the selected trigger provides.

◈ Where the variables come from

The chips are not hardcoded in the dialog. The selected trigger declares them (TriggerType.vars), and the server sends them to the client on the same descriptor that already carries the trigger taxonomy.

Add a new trigger → its variables appear automatically. Zero dialog edits. This is what "any code can fill the event" looks like in the UI.

Why task, not typed inputs, here

The automation carries exactly one value — the file path. A single value rides task cleanly. Typed inputs is for structured per-fire data ({file, connector_id, receivedAt}) — which the file trigger doesn't have.

one value → task  ·  many/typed values → inputs

The author never edits YAML to discover names

Palette lists the tokens. Preview shows the shape. Unknown token (${{trigger.fil}}) → preview renders "" + amber hint. Same chips work in task, a prompt, or any action's payload field.

2 · What the dialog writes — automation.yaml

author output<repo>/.pi/automation/invoice-watch/automation.yaml
on:
  kind: file           # the File trigger
  path: /spool          # static config (from the folder field)
  events: [created]
action:
  kind: flows.run
  payload:
    flow: invoicebot:process
    task: "Process invoice at ${{trigger.file}}"   # ← template; filled per-fire
concurrency: queue         # FIFO — one file = one session (already built)

Static config (folder, flow) is entered once. The ${{trigger.file}} token is the only dynamic part — a placeholder resolved at fire-time, not at save-time.

3 · The per-fire data flow (five hops, three boundaries)

Each arriving file walks this path. The variable bag is the new channel; everything green already exists.

1 · file trigger  (new)
fs.watch("/spool") sees inv-042.pdf land, then fires with a generic variable bag — any trigger fills whatever it knows:
fire({ firedAt, vars: {
  file:     "/spool/inv-042.pdf",
  filename: "inv-042.pdf",
  dir:      "/spool",
  ext:      ".pdf"
}})
2 · automation engine  (un-cut the wire)
Today the fire context is dropped at onFire. Fix = thread ctx through scheduler → runner (queue holds {automation, ctx}) → dispatch.
// engine.buildRunDispatch — ONE central interpolation pass
const payload = interpolate(action.payload, ctx.vars);
//   "Process invoice at ${{trigger.file}}"
// → "Process invoice at /spool/inv-042.pdf"

No action-specific code. Interpolation runs over the whole payload for every action. flows.run stays untouched — it just reads an already-filled payload.

3 · flows.run action  (unchanged)
Builds the cross-boundary event from the (already-substituted) payload:
buildEvent({ payload }) ⟶
  { eventType: "flow:run",
    data: { flowName: "invoicebot:process",
            task: "Process invoice at /spool/inv-042.pdf" } }
4 · emit into the spawned run session
Automation spawns a fresh pi session for this run, then emits the event into it (not a direct call to flows — a third boundary):
ctx.emitEventToSession(sessionId, "flow:run", data)
5 · pi-flows inside that session  (already wired)
A copy of pi-flows runs in the spawned session and listens for the event:
pi.events.on("flow:run", d ⟶
  runFlow(d.flowName, { task: d.task }))

# flow step resolves ${{task}} → the real path
- id: process
  task: Process invoice at ${{task}}
      # → "Process invoice at /spool/inv-042.pdf"

✔ Zero flow.yaml change. The flow already treats task as a file path (contentHash(task)). Next file → fresh ctx → its own run.

4 · task vs inputs — when to use which

PathWhat it sendsChannelStatus
Fresh — automation / file trigger one file path dynamic task via ${{trigger.file}} ← the dashboard fix
Resume — your CLI tool structured {invoice_id} typed inputs → ${{flow.input.invoice_id}} ✔ works today

The automation only ever has one value → it uses task. Structured values → inputs. Both ride the same flow:run event (data.task / data.inputs); pi-flows consumes either.
